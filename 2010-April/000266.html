> -See comments below.
>  
>  
> 1. A fundamental concept in Varkon is "the active module."
> This can never be a MACRO module. I assume there is at
> most one active module. Does Varkon allow a condition
> (analogous to the "zero-document state" of AutoCAD)
> where there is no active module?
> -In GENERIC mode there is always exactly one active module.
> -In EXPLICIT mode there is no active module.
> 2. Unlike other modules, a MACRO is not embedded into
> the database, and can call a (different) MACRO module.
>  
> -Yes. 
> 3. The real point of MACRO modules is facilitating
> something like Model-View-Controller design: a MACRO
> module is a Controller component, the MBS/MBO files
> capture Model components, and a RES file is like a View
> component.
>  
> -Yes.
> 4. A non-MACRO module can't (and shouldn't) directly
> invoke a MACRO module. For example, take M to be
> the alert-box example in the page macro-dialog.htm of the
> Varkon Application Developers Manual, and suppose a
> non-MACRO module NM encounters an error condition.
> The "wrong" approach would be for NM to pop up a
> dialog got by re-writing M as a non-macro module.
> The "right" approach would be for NM to create a user
> interface item such as a button that upon clicking would
> show the alert-box dialog.
> Note that I am not finding the above (#4) example
> very convincing. Am I wrong not to be convinced? Can
> someone provide a better example?
> -How you want to use MACRO modules is up to you. If
> you can get what you need in a different way than the
> book says, that's ok with me. If you don't know how to
> do it, read the book.
