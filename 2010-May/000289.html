Is there a way for a module making a sequence of
get_next_gm() calls to save its place between calls?

My actual immediate concern is whether I might
be missing a way to tidy up my code, though
there are run-time consequences.

Suppose the code considers all pairs (e1,e2),
where each ei ranges over a set Si of entities.
Perhaps the code finds all intersections between
circles (S1) and rectangles (S2).
My code saves a REF c to the last circle retrieved;
then after dealing with all pairs (c, e2) it re-starts with
set_root_gm(#0) and comes forward watching for c,
and takes the circle after c.
If the GM has n entities, this means the code is
making about 0.5*n2 get_next_gm() calls looking
for circles, instead of about n calls as might (naively?)
be wished. That is in comparison with about |S1|*n
calls looking for rectangles -- so if circles are sparse
in the GM, the running time becomes dominated by
re-locating the previous circle.

Of course we could also save the parent of c,
but I think that complicates without essentially
changing the above.

In the analogy of the GM to a Unix filesystem,
set_root_gm() is like setting the current working
directory, whereas I'm wondering about perhaps
a set_current_gm() that would be like setting a
directory or file or pipe as a point from which to
search forward.
