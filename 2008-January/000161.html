> soren.larsson at tech.oru.se writes:
>
>> Now to my question. In the varkon start window you can select to open
>> a module (mbo), which would then be regarded as active module. But
>> what if you only have this as mbs?
>>
>> I would prefer also to have the option to open an mbs-file as active
>> module. The first thing that I would expect to hapen is the
>> compilation of this to MBO, and then continue as usual.
>>
>> Maybe (i'm hovever not sure if it is a good idea) the file dialog
>> filter could do this automatically by showing all filenames (without
>> extention) that have MBO and/or MBS extensions. If MBO exist this
>> would be first priority to open. If MBO does not exist, the MBS would
>> open.  (or maybe this will comfuse those who are trying to understand
>> what is going on).
>
> And if there is a problem reading the MBO file because of a
> incompatible file format or whatever Varkon could fall back to
> compiling the MBS.
>
>>
>> Another even more drastic approach would be to se the active module
>> mbo file as a temporary cache during execution only (at least for the
>> active module). Saving the job would then save only MBS file and open
>> a job would always start by compilation. Starting would be some
>> slower, but during work there would be no difference.
>
> I really like this idea.  The cache could be kept between Varkon
> sessions.  If Varkon reads a MBS file and finds a MBO file in its
> chache that is newer than the MBS file it could use the MBO file.
> Otherwise it has to recompile the MBO.
>
>>
>> Anyhow, if it was easy ore even standard behavior to open MBS as
>> active module, you could after you have prepared an example for
>> posting to varkon org, strip your folder from all res and mbo -files
>> and. Thereafter create zip-file. The package would then be small and
>> portable.
>
> And if MBO files were outside of the project folder it would be very
> easy to share a project across the network.
