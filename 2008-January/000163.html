Hello Johan,
> 3. One of the reasons for a compiler is to translate the
>    code into something which is more effective to execute
>    than plain text. Comments are not exeutable and most
>    compilers strip comments out of the binary code they
>    produce. So does the MBS compiler. It has been
>    proposed that we change this behaviour so that comments
>    are preserved during compilation. That would mean that
>    MBO files would become larger (comments need space) and
>    also that the decompiler would have to reformat coments
>    the same way it does with code today. Would this be
>    acceptable you think ?
>   
One way to achieve this would be, to store a flag for whence the MBO
file came, MBS or GUI and never directly overwrite user edited MBS.
Instead a 3-way merge could be done, so the comments get back into
place.
If the user deletes a block in the gui containing comments in the edited
MBS, this will create a slight problem, but it couldn't be solved by
keeping the comments in MBOs, because you need some sort of
anchoring of comment to code all the same.
Unless the user follows a know convention you will always have the
problem what comment belongs to what code.

Let's assume, the user added a statement at the bottom by hand and
commented it. Then he deletes said statement in the GUI and adds
something else. How would you tell, that this - say - pre-comment
is not a post comment to the previous statement and avoid
the new last statement to the old comment?
Ev. you need to store the whole thing as diff-chunks => that can't
be edited directly and you and up with an internal revision control
system...

Regards, Armin
