Hi all,

>> As I
>> understand it they are normally not present in a Varkon project and
>> Varkon decompiles the MBO file each time you want to edit the active
>> module.

Yes this is right. The interactive GUI works directly on the
binary representation of a module (MBO format). To get the
corresponding MBS code decompilation is required.

I have three comments:

1. menu/file/export/MBS (f144) will export your module on
   MBS format (file selector is used to specify the
   name and location). Same function is also available
   as menu/file/save_as/MBS.

2. When you save or save_and_exit Varkon by default
   saves the module in MBO format. These two functions
   also exist in a version that saves the MBO and decompiles
   and also saves the MBS automatically. To replace the two
   current functions with their MBO+MBS versions is just
   a matter of a simple edit of the menu file. Would it
   be better to have this as default ? All users would
   then automatically get 3 files per job. MBS + MBO + RES
   (There is also the JOB file but it will be removed
   in Varkon 2.0).

3. One of the reasons for a compiler is to translate the
   code into something which is more effective to execute
   than plain text. Comments are not exeutable and most
   compilers strip comments out of the binary code they
   produce. So does the MBS compiler. It has been
   proposed that we change this behaviour so that comments
   are preserved during compilation. That would mean that
   MBO files would become larger (comments need space) and
   also that the decompiler would have to reformat coments
   the same way it does with code today. Would this be
   acceptable you think ?
